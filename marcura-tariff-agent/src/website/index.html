How it works

From two PDFs to an itemised, traceable bill — five stages, all dynamic.

Component (data flow)
Critical-path step
Cache · indexer

Cold run: ~3 min · Hot run: ~17s. Caches mean re-parsing the same tariff PDF or re-extracting the same rule set is instant. Per-rule calculation is the dominant cost — parallelised, batched, with thinking-mode off, so it stays fast and cheap.

Built on

Gemini 2.5 Flash

The reasoning core. Reads the tariff to extract rules, then interprets each rule per vessel to produce the formula and values. Used with thinking_budget=0 for per-rule batched calls (cheaper, deterministic on structured tasks).

langchain-google-genai

Thin LangChain wrapper around the Gemini API for chat-message dispatch and async invocation. Swapping providers means changing one constructor.

LlamaParse Cloud

Document parser that turns scanned tariff and certificate PDFs into structured Markdown — preserves tables, footnotes, and headings the LLM needs to read accurately.

simpleeval

Sandboxed arithmetic evaluator. Receives the LLM's free-form formula string and computes the number deterministically. No Python eval, no LLM math.

FastAPI + asyncio

Async REST API framework. The per-rule calculator dispatches batches in parallel via asyncio.gather + a 10-wide Semaphore — ~32× faster than sequential.

Pydantic

Schema validation for every domain object — VesselProfile, ExtractedRule, RuleStore, CalculatedCharge — so a malformed LLM response is caught at the boundary, not deep in the calculation pass.

Content-hash disk cache

Plain on-disk cache keyed by SHA-256 of the input bytes (PDFs) or parsed text (rules). No Redis, no infrastructure — re-runs of the same tariff are instant.

Docker

Multi-stage container for reproducible deployment to any platform that runs containers — local laptop, Cloud Run, ECS, Kubernetes.

The HarbourMind build journey

Building HarbourMind wasn't a straight line. The first version silently dropped most of the tariff. The fast version was wildly wrong. The accurate version cost too much. Each round of testing exposed a new failure mode and forced a clearer architectural decision. This is what that journey looked like, in plain English.

01

The problem we started with

We started with a logical assumption: tariffs have patterns. We built five types of charge calculators and asked our AI to fit every rule into one of these pre-defined boxes.

It looked perfect. Then it failed. The system found 97 rules in the Durban tariff but only 10 actually worked. The other 87 silently broke because they didn't fit the boxes we'd created for them. The final bill came back at R 18,097 instead of R 580,000. The system looked fine — no errors, clean output. It was just under-billing by 97%.

The lesson: you can't force tariff rules into rigid boxes when you're trying to handle thousands of different ports, each with their own way of writing rules.

02

The pivot — from rigid boxes to flexible formulas

We tried three approaches:

  • Let the AI write the code — Have it generate formulas and run them. Too risky and expensive to secure properly.
  • Have the AI do all the math — AI reads tariffs beautifully, but it's terrible at arithmetic. Numbers kept coming out wrong.
  • Split the jobs — AI reads the tariff and extracts what the formula is (like "base fee plus tonnage rate"), and let software do the actual math.

We chose option 3. Humans are great at interpretation. Computers are great at math. Do both. The AI now reads "Port dues = base fee per 100 tons, plus rate per 100 tons per day", hands us the formula and the numbers, and software plugs them in. The bill jumped from R 18K to R 564K immediately.

03

The three answers — beyond "yes or no"

Early versions gave two answers: compute the charge, or skip it. Simple. But reality isn't binary. A rule might say "add a surcharge if the service happened outside normal working hours". The AI reads this. But the shipping certificate doesn't say what time the service happened.

Three options: compute it anyway (over-bill), skip it (under-bill), or admit we don't have enough information (honest, but unconventional). We chose the third.

Now the system gives three answers: charges we're confident about (we have the data and know how to calculate), charges that clearly don't apply (the rule doesn't fit this vessel), and charges that need clarification (the rule looks right, but we're missing information).

04

The stacking trap — when similar rules fight

One day, the system calculated 11 different cargo charges at once. Cargo dues depend on how the cargo is routed — overseas, domestic coastal, or transferred between ships. Each route has different pricing. The tariff describes them as options: pick one based on how your cargo moves.

But when we asked the AI to evaluate each rule separately, all 11 looked equally valid. The system calculated all of them. The bill tripled.

The fix: before asking the AI to calculate a rule, show it all the competing rules at the same time. Tell it: "only one of these applies. Which one?" Suddenly the AI understood the context. It picked the right rule. The over-billing stopped.

The lesson: a smart system evaluating one rule in isolation is worse than a reasonably smart system evaluating all related rules together. Context matters.

05

The reproducibility crisis — eliminating guesswork

We ran the same calculation three times and got three different answers: R 481K, R 509K, R 565K. That 17% swing is unacceptable for billing. The AI was making judgment calls differently each time:

  • Time period — rules mention "days in port" and the certificate shows both 3.39 days alongside and 7 calendar days in port. Which one? The AI kept picking differently.
  • Multiplying by operations — pilotage costs per service, and the certificate lists 2 operations. Multiply by 2? Sometimes yes, sometimes no.
  • Rounding — should 3.39 days become 3.5 for calculation? We had no clear rule.

We removed every ambiguity with simple, firm decisions. Per-service charges always multiply by the number of operations. Use the exact day value provided (3.39), never round up. When both time periods exist, use the more specific one.

Result: three consecutive runs returned exactly R 564,840.07 each time. Identical to the cent.

06

The speed journey — making it practical

The original version extracted rules one at a time and calculated one at a time. With 112 rules, that was 224 separate requests. It took 9 minutes. Three changes brought it to 17 seconds:

  • Parallelize — ask for multiple rules at the same time instead of waiting for one before asking for the next. Cut time in half.
  • Cache — if we've already extracted rules from the Durban tariff, don't extract them again. Reuse the answer.
  • Batch — instead of asking for one rule at a time, send five related rules in a single request. Get five answers back.

Final time: 17 seconds. The original was 547 seconds. 32× faster. The remaining seconds are network round-trips and parsing — there's not much left to optimize.

07

The cost story — smarter, not just cheaper

We picked a powerful AI model and discovered it was costing way more than expected. The model was doing internal "thinking" — working through logic behind the scenes — and we were billed for all of it.

We tried a cheaper model. It started making mistakes on complex rules. We couldn't sacrifice accuracy for cost.

The solution: keep the powerful model, tell it to skip the internal thinking, and just give us the answer. Once rules are batched properly (showing related rules together), the AI doesn't need that extra reasoning — it already has the context.

Cost dropped about 10× with no loss in accuracy. We were paying for thinking we didn't actually need.

08

The calibration — honest within 3%

Marcura's reference total for SUDESTADA at Durban: R 582,855.45. Our system's calculation: R 564,840.07. Difference: 3.1%.

This gap isn't a math error — it's about how tariffs are interpreted. Port invoices traditionally split into a vessel invoice (port dues, pilotage, tugboat charges, light dues) and a cargo invoice (charges for handling the cargo). Different parties pay each — the vessel operator pays the first, the cargo owner pays the second.

Marcura's reference includes both. Our system, without explicit information about what cargo is on board and where it's going, marks cargo charges as "needs clarification". We could guess, but guessing on a cargo invoice is worse than asking. The 3.1% gap is the right behaviour: transparent about what we don't know, rather than precise about things we're guessing on.

09

The hardcoding audit — nothing locked in

We audited every line of code for hardcoded values. Here's what we found:

  • Zero hardcoded rates or fees — not a single number is baked in.
  • Zero hardcoded port names — everything comes from the uploaded tariff.
  • Zero assumptions about rule types — rules extract their own formulas.

What is built into the system: universal maritime billing principles. "Port dues are mandatory when a vessel enters." "Light dues apply on first arrival." "Pilotage surcharges only trigger on specific conditions." These aren't Durban-specific. Every port in the world uses the same billing principles.

This is why the system works for any port. Change nothing in the code. Upload a Singapore tariff or a Rotterdam tariff. It will extract and calculate correctly.

10

What the AI actually understood

We asked the AI to extract rules from the Durban tariff. It found 112 distinct charges, reading the document with real sophistication:

  • Compound formulas — port dues aren't one number, they're a starting fee plus a daily charge. Both extracted and combined correctly.
  • Tiered pricing — for a 51,300 GT vessel, towage depends on the size bracket. The system found the right bracket.
  • Mutually exclusive options — light dues has 4 different schedules depending on vessel type. The system recognised these as choices, not stacking charges.
  • Commodity-specific pricing — cargo dues for "Iron Ore" differ from generic dry-bulk. The system looked up the specific commodity.
  • Exact fractional days — the tariff says to pro-rate fractional days. The system used 3.39 days exactly, not rounded.
  • Honest uncertainty — when rules referenced external regulations not in the tariff (like SAMSA Levy rates), the system marked them "needs clarification" rather than guessing.
11

What a shipping certificate doesn't tell us

A shipping certificate is intentionally thin — it identifies the vessel, not the whole transaction. This creates honest gaps:

  • Cargo routing — foreign-going, domestic coastwise, or transhipment? Cargo dues depend on this. The certificate doesn't say.
  • Service timing — what time did the pilot come aboard? Were tugboats called during night hours? The certificate gives dates, not times.
  • What actually happened — did the vessel arrive late? Was the crew unprepared? The certificate is static, it doesn't capture the day's events.
  • Invoice structure — how many parties will be billed? Are there account-splitting fees? Not in the certificate.

A real port billing system would combine the certificate with operational logs (from the port), cargo manifests (from customs), or a quick conversation with the customer before finalising the bill. We don't pretend the certificate tells the whole story. For everything else, we say: "I can calculate this, but I'm not sure about the inputs. Here's what I need you to confirm."

12

Why this matters — the trust layer

Billing is trust. A shipping company, port authority, and cargo owner all need to look at an invoice and understand where every number came from.

A system that just outputs a number and says "trust me" is a liability. A system that shows its work — that says "here's the rule from the tariff, here's the data from your certificate, here's the formula I used" — is one people can actually rely on.

HarbourMind is built on this. Every charge is traceable. Every skipped rule has a reason. Every gap in the data is surfaced, not hidden. That's the real difference. Not just accurate calculations, but transparent calculations.

Future-proofing the build

The architecture stays valid as tariffs, vessels, ports, and even LLM providers change. Concretely:

  • Zero numeric hardcoding — no rates, fees, port names, or vessel data baked in. Everything comes from the uploaded PDF.
  • Free-form formulas — the calculator accepts any arithmetic expression the LLM produces. New tariff structures need no code change.
  • Universal billing conventions only — the prompts encode patterns that hold for every port (mandatory charges, surcharge triggers, drydock defaults), not Durban specifics.
  • Provider-swappable LLM — one constructor file. Swap Gemini for GPT or Claude with no logic changes.
  • Three-state output — handles any data gap a thin certificate creates, indefinitely.
  • Content-hash cache — re-processing the same PDF or rule set is free and stays free across deploys.
